# LGTM Dogfood Retro — 2026-07-11

## Outcome

LGTM completed seven historical Python-task replays across five hook events, shipped an audited waiver path, and released verified Linux and macOS binaries as `v0.1.2`. The replay preserved the dogfood repository's working tree and exercised clean, failing, security, malformed-input, and bounded long-command cases.

## What Worked

- Deterministic replay exposed policy behavior without claiming historical tests passed.
- Stop remained the authoritative hard gate; PostToolUse gave timely feedback.
- Evidence distinguished failures, missing tools, timeouts, overrides, and waivers.
- Immutable failed release tags preserved an honest deployment trail before `v0.1.2` passed.

## Friction and Corrections

- Cumulative test rules blocked too early while tests were still being edited. PostToolUse now warns; Stop enforces the finished slice.
- Repository command detection chose bare `pytest` in a uv project. Detection now emits `uv run pytest`.
- A fixed 30-second command limit misclassified a real four-minute suite. Repositories can set bounded command-specific timeouts.
- Repeated broad-exception findings at legitimate best-effort boundaries proved waiver pressure. Waivers now require rule, reason, owner, and future expiry; protected security rules remain unwaivable.
- Path normalization bugs appeared only on macOS release runners. Deleted paths and filesystem aliases are canonicalized before comparison.

## Lessons Fed Back

Build workflows should separate fast advisory feedback from final enforcement, state what a compressed acceptance gate cannot prove, preserve failed release attempts, and make the reviewer independent of the executor without blindly invoking an executor-specific review tool. These rules were added to Nautilus Phase 4 guidance.

## Follow-up

Keep missing-tool output concise, validate emitted artifacts against schemas, and retain malformed-input tests for every hook. Revisit telemetry only after broader adoption creates a real measurement need.
